## 3. 模块设计

### 3.1 auth

| 类/文件 | 说明 |
|---------|------|
| `AuthService` | 注册、登录、签发 access/refresh |
| `JwtStrategy` | Passport JWT 解析 `sub` → userId |
| `AdminAuthService` | 后台账号校验，角色写入 JWT `role` |

密码：`bcrypt`，cost=10。

### 3.2 catalog

| 能力 | 实现要点 |
|------|----------|
| 类目树 | 递归查询或一次拉平按 `parentId` 组树；`path` 在创建/移动时更新 |
| SPU 列表 | C 端可见 `ON_SALE` / `SOLD_OUT` / `RESTOCKING`；支持 `categoryId` 筛选 |
| SPU 详情 | 同上可见状态；联表 SKUs；`minPrice = MIN(sku.price)` |
| SPU 状态 | `NOT_LISTED` 未上架、`ON_SALE` 已上架、`SOLD_OUT` 已售罄、`OFF_SALE` 已下架、`RESTOCKING` 补货中；库存变更后自动同步（见共享约定） |
| 管理端 CRUD | 事务内创建 SPU + 批量 SKU；规格矩阵由前端传入 `skus[]` |

### 3.3 cart

- 表：`cart_items(user_id, sku_id, quantity, selected)`，唯一索引 `(user_id, sku_id)`。
- `GET /cart`：联 SKU/SPU，计算 `valid`（非已上架、售罄、补货中或 `stock=0` 时失效）。
- 加购：数量上限 `min(99, stock)`。

### 3.4 inventory（核心）

**服务接口**：

```typescript
interface InventoryService {
  checkAvailable(skuId: number, qty: number): Promise<boolean>;
  reserveForOrder(orderId: number, items: { skuId: number; qty: number }[]): Promise<void>;
  releaseByOrder(orderId: number): Promise<void>;
  confirmByOrder(orderId: number): Promise<void>; // 支付成功：清 reserved
}
```

**两阶段 SQL（示意）**：

```sql
UPDATE skus
SET available = available - :qty,
    reserved = reserved + :qty,
    version = version + 1
WHERE id = :skuId
  AND available >= :qty
  AND version = :version;
```

- `affected === 0` → 抛 `40902`。
- 热点 SKU：方法入口 `redis.set('lock:sku:'+id, '1', 'NX', 'EX', 3)`。

`inventory_reservations` 表：`order_id, sku_id, quantity, expire_at`，便于对账。

### 3.5 order

**OrderService.create**（事务）：

1. 校验地址归属用户。
2. 解析商品行（购物车 ids 或 directBuy）。
3. `inventory.reserveForOrder`。
4. 生成 `orderNo`（雪花或 `日期+随机`）。
5. 写 `orders`、`order_items` 快照，`status = PENDING_PAY`。
6. 写 `order_status_logs`。

**OrderStateMachine**：

```typescript
const ALLOWED: Record<OrderStatus, OrderStatus[]> = {
  PENDING_PAY: ['PAID', 'CANCELLED'],
  PAID: ['SHIPPED', 'REFUNDING'],
  SHIPPED: ['COMPLETED', 'REFUNDING'],
  REFUNDING: ['REFUNDED', 'PAID', 'SHIPPED'],
  // COMPLETED, CANCELLED, REFUNDED 无出边
};
```

`transition(orderId, to, operator?)` 内校验边 + 写日志 + 副作用（库存/支付）。

### 3.6 payment

| 步骤 | 说明 |
|------|------|
| create | 校验订单 `PENDING_PAY`、金额一致；写 `payments`；返回 `payUrl` |
| notify | 验签 → Redis 幂等 → 事务：payment SUCCESS + order → PAID + `inventory.confirmByOrder` |
| mock-notify (admin) | 内部调用 notify 逻辑，供 QA |

**模拟收银台**：`GET /mock-pay` Controller 返回简单 HTML，表单 POST 到 notify 路径（带签名）。

### 3.7 jobs

| 任务 | Cron | 逻辑 |
|------|------|------|
| order-timeout | `*/5 * * * *` | `PENDING_PAY` 且 `created_at < now - 30min` → CANCELLED + release |
| auto-complete | `0 3 * * *` | `SHIPPED` 且发货超过 15 天 → COMPLETED |

使用 `@nestjs/schedule` 或 BullMQ（轻量用前者）。

## 4. Prisma 模型要点

与架构说明书第 11 节一致，补充字段：

```prisma
model Sku {
  id        Int      @id @default(autoincrement())
  spuId     Int
  specsJson Json
  price     Int      // 分
  stock     Int      // available 可售
  reserved  Int      @default(0)
  version   Int      @default(0)
  status    String   @default("ENABLED")
  @@index([spuId])
}

model Order {
  id          Int         @id @default(autoincrement())
  orderNo     String      @unique
  userId      Int
  status      String
  totalAmount Int
  payAmount   Int
  @@index([userId, status])
}
```

## 5. 中间件与横切

| 组件 | 行为 |
|------|------|
| `TransformInterceptor` | 成功响应包 `{ code:0, data, requestId }` |
| `HttpExceptionFilter` | 映射异常 → 业务 code |
| `LoggingInterceptor` | 记录 method、path、duration、requestId |
| `ValidationPipe` | `class-validator` DTO 白名单 |

**requestId**：中间件生成 UUID，放入 `AsyncLocalStorage`，日志与响应共用。

## 7. 开发任务拆分（服务端）

| 序号 | 任务 | 验收 |
|------|------|------|
| S1 | 工程初始化、Prisma schema、migrate | 表创建成功 |
| S2 | auth C 端 + admin 登录 | Postman 拿到 token |
| S3 | catalog 类目 + SPU/SKU CRUD | 管理端 API 可调 |
| S4 | cart CRUD | 加购合并数量 |
| S5 | inventory + order.create | 并发下单不超卖（脚本压测） |
| S6 | payment + mock 回调 | 订单变 PAID |
| S7 | ship / cancel / refund | 状态机边全覆盖 |
| S8 | 定时关单 | 超时订单 CANCELLED 且库存回滚 |
